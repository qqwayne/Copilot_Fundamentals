# Emergency Log Triage デモ

## 🚀 作成するもの

この短いラボでは、GitHub Copilot を使って**最小限のスケルトン（雛形）**から、動作する Python ユーティリティへと作り上げます。完成するツールは次のことができます：

- 普通ファイル形式（`.log`）または gzip 圧縮（`.log.gz`）されたアクセスログを**ストリーミング処理**
- 直近N分以内のログだけを取り出す
- ステータスコードとURLの組み合わせごとに回数を集計する
- コンパクトな **Markdown 風ヒストグラム**（ランキング表）を出力

このラボの目的は「完璧なコード」を書くことではありません。\
**Copilot の UI と自動補完の“魔法”を体験すること**がゴールです。コメントブロックから Copilot にプロンプトを与え、提案を受け入れ（または微調整し）、数秒で “それっぽい” 本番品質のボイラープレートが生成される様子を観察します 🪄

## 🧑‍💻 ハンズオン手順

### 1. クローン & オープン

```bash
git clone https://github.com/qqwayne/Copilot_Fundamentals.git
cd demos/copilot-essentials/
code .
```

---

### 2. 雛形を確認する

`triage.py` を開きます。各関数の中身は `pass` のみで、直前に短い Copilot プロンプト（トリプルクォート）が書かれています。

例：

```python
def read_lines(file_path: Path) -> Iterable[str]:
    """Open plain or gzipped log file and yield each line (stripped)."""
    pass  # ← Copilot がここを埋めます
```

---

### 3. Copilot を呼び出す ✨

1. 関数全体を選択（または `pass` にカーソルを置く）
2. `Tab`（または Copilot の「提案を承認」ショートカット）を押す
3. Copilot が実装を挿入 → 内容をレビューして受け入れる

以下の関数でも同様に繰り返します：\
`parse_line`, `triage`, `render`, `main`

💡 **ヒント**：Copilot の差分を読むことを推奨します。\
「なぜこの正規表現にしたの？」\
「エラーハンドリングはどこ？」\
などを Copilot Chat で質問してみましょう。

---

### 4. スクリプトを実行する

```bash
python3 triage.py --file sample_access.log.gz --minutes 15 --status 499,321 --top 10
```

想定される出力（抜粋）：

```
Rank | Status | Path               | Hits
-----|--------|--------------------|-----
 1   | 499    | /api/v1/users      | 9
 2   | 321    | /login             | 4
 ...
```

- `--status` を省略すると、すべてのステータスコードが対象になります
- `--minutes` や `--top` を変えて、見える結果の粒度を調整してください

---

### 5. いろいろ試す（Experiment）

- プロンプトを変更して「CSV 出力」を追加し、Copilot を再実行
- ログフォーマットを数行だけ壊して、`parse_line` が不正行をうまくスキップできるか確認
- 自分の開発環境の実ログに差し替える（`.log` / `.log.gz` のままでOK）

## 🔍 `sample_access.log.gz` の中身

- **形式**：Apache/Nginx の “combined” アクセスログ  
  例：`ip – – [timestamp] "METHOD path HTTP/x.y" status size ...`
- **時間範囲**：直近 60 分（実行ごとにランダム化）
- **ステータスコード**：通常の 2xx / 3xx に加えて、400 / 404 / 499 / 500 / 503、そして少数のカスタム 321
- **エンドポイント**：典型的な REST ルート（`/api/v1/*`）、静的アセット、ヘルスチェック等

🗒️ **なぜ gzip？**  
実システムではログがローテーションされ、圧縮されるのが一般的です。初心者のうちに、両方を透過的に扱えるようにするためです。

## 🎯 学習目標

| スキル | 何で示すか |

| ログ処理の定石 | 遅延ストリーミング、正規表現パース、`collections.Counter` |
| CLI の基本 | `argparse` と分かりやすい `--help` |

## 💬 よくある質問

**Q. “なぜ pandas を使わないの？”**  
A. 標準ライブラリだけで完結させ、インストールが学習の流れを止めないようにするためです。

**Q. “Copilot は最初から常に完璧？”**  
A. いいえ。ガイドが必要です。カーソル位置を変える、プロンプトを具体化する、最初の 1 行だけ自分で書く、などで結果が改善します。

**Q. “ログはどれくらい大きくても大丈夫？”**  
A. 1 行ずつ読み込むため、メモリ使用は一定です。制限は基本的にディスク容量です。
